What is Astropy?¶

Once upon a time ...¶

Independent projects existed around the world to help astronomers used Python for their daily tasks. These tasks included

  • Handling a variety of formats of tabular data.
  • Dealing with FITS files.
  • Dealing with World Coordinate Systems.
  • Do cosmological calculations.
  • Invoke Virtual Observatory based queries. and so on ...

Different developers, different teams, different interfaces.¶

Relatively poor interoperability.

Enter Astropy!¶

Astropy is a community project that aims to unify these interfaces together into one common package useful for all astronomers.

Affiliated Packages¶

These packages are an independent set of projects and thus not a part of the Astropy project. But the developers of these projects share the same vision of a unified interface/approach to doing things. And thus they sign up their packages as affiliated packages.

Look at: http://www.astropy.org/affiliated/index.html for more details.

What Astropy can do?¶

Lots!

But we will attempt to cover here basics of a subset of features.

  • Tables
  • FITS files
  • World Coordinates/Coordinate Systems
  • Modeling
  • Cosmology

Tables¶

For low level access to reading ASCII tables, one can use the classes in the sub-module astropy.io.ascii. This is seldom required. We instead use astropy.table which is a higher level and more convenient API to dealing with tables. It is only when the higher level API has trouble loading your file, it might benefit you to move to the lower API and play around with it.

So, let's read out a simple table.

In [1]:
# First, import the table class.
from astropy.table import Table

# Next, use its read method to read a table
demo_table = Table.read('demo.txt', format='ascii')

That's it. The table is loaded. Before we move further, couple of points.

  • format="ascii" is needed for all ASCII files since they can have a variety of extensions such as .dat, .txt, .csv etc.
  • For files with extensions such as .vot, .xml (VOTables), .fits (FITS Tables) etc. the format need not be given.
  • If an ASCII file does not load, it means the tables violate some assumptions in terms of structure. In this case, refer to documentation of astropy.io.ascii and try passing various flags to help Astropy interpret the file.

Dealing with FITS files¶

Recall that FITS stands for "Flexible Image Transport System". It is a file format designed by astronomers. And it is indeed flexible - it can store images (as 2-d arrays) and it can store tables (ASCII and binary). And a given file can have both - this allows one to attach tabular data directly to images. eg. a catalog of photometry can be attached to the image from which it was extracted.

Recall that a FITS file is made of one or more header/data units (HDUs).

  • Each HDU is made of a header unit which contains metadata in the form of a key-value pair. This part is compulsory.
  • An optional component within a HDU is the data unit which contains the data unit.

Astropy has a sub-module for handling FITS files. Let's see how to load a file.

In [3]:

# To load a FITS file

World Coordinate Systems¶

If you open an image is ds9 or any other image viewer, you will notice that as the mouse pointer moves across various regions in the image, a part of the window will reflect the current coordinates in some coordinate system. This is possible because the image header contains a whole bunch of keywords which can be used to compute the coordinates as a function of the pixel coordinates.

What one needs is the coordinate for one pixel and then variation in coordinates for every pixel of movement. But if the image is very large, the curvature of the sky can become important. Then additional information to account for second order effects will need to be recorded

Let's see how a good fraction of all this is handled using Astropy.

Simple steps:

  • Get hold of the header.
  • Pass it to a WCS object constructor.
  • And use this object for coordinate conversions.

Modeling¶

This is a library in Python designed to give you

  • Access to commonly used models.
  • As well as fit them to various data.

Let's see how to simply use models.

We saw how trivial it is to use a model and actually evaluate it over a range. But a more useful thing we need to do with models is to fit some data. Now, pretend that x and y_obs are two arrays that contain our observed data. We can start by plotting the same.

In [89]:
plt.scatter(x, y_obs)
Out[89]:
<matplotlib.collections.PathCollection at 0x7f1c48f2d9b0>

Once we have a plot, our next step is to choose a model. What model shall fit the data well? Let us assume that these are spectral features with some known wavelengths. So, we may assume that each feature is a Gaussian. To minimize number if independent parameters, we may also consider that the difference in the wavelengths is a constant. Such a constraint obviously comes from our prior knowledge of the physical system that generated the spectrum.

Next, these emission features clearly are not on their own - they are stop a continuum. Assuming that the continuum is not varying at a fast rate wrt wavelengths (X-axis), we can further assume that a quadratic polynomial suffices in accounting for this.

So, our model is a second order polynomial plus two Gaussians, with different means but separated by a known amount.

In [91]:
# So, let us construct our model.
model = models.Gaussian1D(amplitude=1.0, mean=12.0, stddev=0.5) +\
        models.Gaussian1D(amplitude=1.0, mean=13.5, stddev=0.5) +\
        models.Polynomial1D(degree=2)

def constraint_mean(model):
